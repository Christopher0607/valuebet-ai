# ValueBet 本地版 — FastAPI + SQLite + React

一个完全跑在你自己电脑上的系统。没有 Supabase 账号，没有云服务依赖，数据存在本地的 `valuebet.db` 文件里。

**模型现状**：Dixon-Coles 双变量泊松模型，进攻力/防守力参数来自对 32,372 场真实历史比赛（1990年至今）做最大似然估计（MLE）拟合，不再是早期版本那种 ELO 差值的线性近似。每场比赛结束后，双方球队的参数会通过贝叶斯增量更新（泊松-伽马共轭）自动微调，持续跟着最新战绩变化，不需要重新跑一次全量训练。世界杯数据上的真实回测结果目前在 **准确率67-68%区间，RPS 0.14-0.15区间**（随机猜测基准是 0.245，数字越低越准）——具体数字会随着新比赛不断打完持续小幅波动，这是未经美化的真实结果，不是一个固定不变的数字，打开「回测」页面看到的永远是当下最新的真实值。

## 这个"每12小时自动更新"具体是怎么运作的

**必须先说清楚这件事的真实边界，不然你会误以为这是一个装完就能丢在一边不管的东西。**

后端启动时，`APScheduler` 会在同一个 Python 进程里起一个后台线程，每12小时触发一次：抓 openfootball 最新赛果 → 把刚打完的比赛结果折算进对应球队的贝叶斯后验（进攻力/防守力跟着最新表现微调）→ 用更新后的参数重新跑 Dixon-Coles 预测 → 自动结算到期的虚拟盘和实盘注单。这是真实的、会实际执行网络请求和数据库写入的定时任务，而且这次连并发场景都验证过：服务器启动时的自动首次更新和你手动点击的更新如果撞在一起，系统会用锁机制让其中一个诚实地报告"已有更新在进行中，本次跳过"，而不是让两者同时写库产生冲突。

**但这个定时任务只在 `uvicorn` 进程存在的时候才存在。**

- 你关掉运行后端的终端窗口 → 定时任务停止
- 你电脑睡眠或关机 → 定时任务停止
- 你只是把网页关掉，但终端还在跑 → **定时任务继续正常工作**，因为调度逻辑在后端进程里，不在浏览器里

换句话说：**只要那个跑着 `uvicorn` 的终端窗口一直开着，它就会真的每12小时自动更新一次，不需要你去点任何按钮。** 这跟之前浏览器 artifact 版本"标签页一关就停"的情况完全不同——因为这次调度逻辑真的在一个持续运行的后端进程里，不依赖浏览器标签页是否在前台。

如果你想要电脑睡眠、关机之后也能继续跑，看下面「无人值守运行」那一节。

## 快速开始（本地，最简单）

### 第一步：确认 MLE 参数文件存在

模型依赖 `backend/training/fitted_parameters.json`——这是已经跑好的训练结果，随项目一起打包，**你不需要重新训练**就能直接用。如果以后想用更新的历史数据重新拟合，进 `backend/training/` 目录跑：

```bash
cd backend/training
python3 train_mle.py
```

这会读取同目录下的 `historical_results.csv`（32,372场历史比赛），重新生成 `fitted_parameters.json`。正常情况下你不需要做这一步，除非你想调整训练脚本里的超参数（时间衰减半衰期、样本量门槛等）。

### 第二步：后端

```bash
cd backend
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

首次启动会自动：创建 `valuebet.db`、注册世界杯赛事、立即跑一次更新（不用等12小时）、然后开始12小时循环。

### 第三步：前端

另开一个终端：

```bash
cd frontend
npm install
npm run dev
```

打开 `http://localhost:5173`。

## 用 Docker Compose（一条命令起两个服务）

```bash
docker compose up
```

同样打开 `http://localhost:5173`。SQLite 文件会持久化在 `backend/valuebet.db`，容器重启不会丢数据。

## 无人值守运行（关机/睡眠后依然自动更新）

上面两种方式都要求你的电脑一直开着、进程一直在跑。如果你想要真正做到"电脑关了也在后台跑"，有三个选项，从简单到彻底：

**选项A：树莓派或旧电脑常驻**
把 `docker compose up -d`（`-d` 是后台运行）跑在一台你不常关机的机器上（比如闲置的旧笔记本、树莓派），它插着电、连着网，就会一直按12小时循环更新，你的主力电脑关不关都没关系，前端页面通过局域网IP访问它就行。

**选项B：云端小型服务器**
把这套 Docker Compose 部署到一台几十元一个月的云服务器（Vultr、DigitalOcean、阿里云轻量应用服务器都可以），24小时开着，前端做成公网可访问，这是最接近"完全自动、随时随地能看"的方案——但这就是我们最早讨论的 Supabase + Vercel 方向想解决的同一个问题，只是换成了你自己管理服务器而不是用托管服务。

**选项C：系统级后台服务**
Mac 用 `launchd`，Linux 用 `systemd`，Windows 用「任务计划程序」把 `uvicorn` 注册成开机自启的后台服务，这样只要电脑开机（不需要登录、不需要开着终端），它就会自动跑起来；电脑睡眠/关机时仍然会停，这个限制没有办法绕过，因为睡眠关机时整台机器都不执行代码。

如果你想要选项C的具体配置，告诉我你是 Mac / Windows / Linux，我给你写对应的服务配置文件。

| `/api/backtest-summary` | GET | 回测统计 |

## 贝叶斯实时更新是怎么运作的

MLE 训练出的进攻力/防守力参数是"固定不变的历史平均水平"——西班牙的参数不会因为昨天赢了5:0而变化，除非重新跑一次全量训练。贝叶斯更新解决的正是这个问题：每次调度器抓到新的已完赛比赛，会用泊松-伽马共轭公式把这场比赛的真实进球数折算进双方球队的后验分布，让参数持续跟着最新战绩微调，而不需要每次都重新训练。

打开前端后可以在球队详情里看到当前的进攻力、防守力，以及一个不确定性数值（标准差）——场次越多、结果越稳定的球队，这个数值越小，代表模型对这支队伏的判断更有把握；反过来，一支刚打了几场比赛、战绩起伏很大的球队，不确定性会明显更高。

**已知限制**：每场比赛只会被折算进后验一次（有一个 `bayesian_folded_in` 标记防止重复计入，这个问题在开发过程中真实出现过，已经修复并用重复调用测试验证过）。但如果你直接修改数据库里已经处理过的比赛比分，系统不会自动重新折算——这种情况需要手动清空对应球队的 `bayesian_team_states` 记录，让它重新从 MLE 起点开始累积。

## 关于串关（Parlay）——只支持跨比赛的独立事件

`/api/parlay` 只能组合**不同比赛**里的独立事件，比如"西班牙赢意大利"和"法国赢德国"——两场比赛互不影响，联合概率就是两个概率相乘。系统会主动拒绝把同一场比赛的两个盘口凑成一注（比如"西班牙赢"+"西班牙进球少于1.5"），因为这两者本质上是同一个泊松过程的两个不同切面，不是独立事件，直接相乘算出来的概率是错的。这种同场比赛内的组合概率，需要用 `/api/score-distribution/{match_id}` 查真实的联合分布。

### 自动串关推荐（`/api/parlay/suggest`）

前端「🎯 串关推荐」标签页里，勾选几场即将开赛的比赛、输入各自的1X2赔率，系统会自动从中搜出3-8腿的最优正EV组合。

核心设计：**只从正EV的单腿里搜索组合**，负EV的腿在进入候选池之前就被过滤掉。这是刻意针对"把几个强队串起来提高赔率"这种直觉设计的——对独立事件而言，`EV_组合 = Π(1+EV_i) - 1`，任何一条腿是负EV，都会拖累整体表现，不管这条腿看起来多"稳"。而且强队的赔率经常被市场系统性压低（"热门-冷门偏差"，一个博彩市场有实证支持的现象），意味着很多"看起来很稳"的强队独赢盘口，其实本身就是负EV——这个工具会直接把这类腿排除在候选池外，用真实计算结果告诉你这一点，而不是空口说教。

返回结果按三个不同标准各挑一个代表：🏆最高EV、🛡️最稳（命中率最高）、🎯赔率最高（在仍是正EV的前提下）。这三者天然互相冲突，系统不会假装存在一个能同时最大化三者的"唯一最优解"——把真实的取舍摆出来，由你自己根据能接受的风险来选。

串关返回结果里有一个 `risk_ratio_vs_weakest_leg` 字段，展示组合命中率相对于"最弱一腿"打了多少折——比如两条腿单独胜率是55%和60%，串起来整体命中率只有33%，相当于最弱一腿打了六折。这个数字故意摆在明显位置，是为了让"赔率更高看起来更诱人"这件事，不掩盖"命中率被压得更低"这个同时发生的事实。

## 关于欧冠、英超、西甲、意甲、德甲等其他赛事

**当前状态：地基已经打好，还没接通。**

`competitions` 表设计上就支持多个赛事同时运行，世界杯之外的联赛用的是完全不同的数据源结构——世界杯是"一个持续的锦标赛"，英超西甲这类联赛是"按赛季分文件"，新赛季一开始就会有一个新的数据文件出现，8月开季、次年5月结束。

已经写好并测试过的部分：

- `guess_current_season()`：根据当前日期猜测应该抓哪个赛季（比如7月份应该猜下一个赛季，但新赛季文件在8月中旬开季前可能还没发布）
- `resolve_season_url()`：先试猜到的赛季，如果文件还不存在（HTTP 404）自动退回上一个完整赛季，避免调度器在休赛期每次都空手而归
- 比分格式兼容：欧洲联赛的历史数据里发现过两种不同的比分JSON格式混用的情况（`{"ft": [2,1]}` 和裸数组 `[0,0]`），已经验证清楚后者不是"未开赛"而是真实的0比0战平，两种格式都能正确解析

还没接通的部分：`competitions` 表里还没有为这几个联赛插入对应的记录，`fetch_results`/`fetch_upcoming` 也还没有区分"世界杯类型的数据源"和"分赛季类型的数据源"该用哪套解析逻辑。这是下一步要做的工作，不是"改一行 `is_active` 就能用"这么简单——上一版文档这里写得过于轻巧，这次改成准确描述实际状态。

## 关于自动化下注 — 这个系统不做，也不会做

「实盘」功能是让你**下完注之后回来手动登记**，帮你算EV、追踪盈亏。这个系统不会、也不应该自动登录博彩账户替你下单：

- 几乎所有博彩平台的服务条款明确禁止自动化脚本下注，一旦触发就是封号甚至没收余额
- 把一个约68%准确率的模型直接接到自动扣款的真实账户，是完全不同级别的风险——模型的不确定性应该由你亲自把关每一注，而不是交给一个后台任务自动执行

如果你在别处看到有人说能做"全自动下注机器人"，建议保持怀疑。

## 已知限制

- **SQLite 是单文件数据库**，适合个人使用，不支持多人同时高并发写入。这套系统本来就是给你一个人用的，够用。
- **模型准确率目前在67-68%区间，RPS 0.14-0.15区间**（世界杯数据实测，随比赛推进会小幅波动），比随机猜测（33%/0.245）好不少，但仍有明显误差率，回测页面的数字是真实、未经美化的结果。
- **调度器不做失败重试**：如果某次12小时更新因为网络问题失败了，会等到下一个12小时周期再试。`/api/update-log` 能看到每次运行的成功/失败记录。
- **贝叶斯更新目前只支持"每场比赛只折算一次"**，如果你手动改数据库里的历史比分，系统不会自动重新计算，需要手动清空对应的贝叶斯状态记录（见上文"贝叶斯实时更新是怎么运作的"一节）。
- **多联赛扩展（英超/西甲/意甲/德甲/MLS）目前处于半成品状态**：`updater.py` 里已经写好并测试过赛季自动探测逻辑（`guess_current_season` / `resolve_season_url`，处理"7月还没到新赛季文件"这种情况），但还没有接入 `competitions` 表和调度流程，这是下一步要做的事。
